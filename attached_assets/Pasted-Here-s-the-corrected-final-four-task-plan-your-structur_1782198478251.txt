## 1 ‚Äî Foundation: Supabase + Express Backend

**What & Why**
Replace the old Replit-managed api-server wiring with a self-contained Express backend and a Supabase integration, providing the shared foundation every other upgrade task depends on: auth client, database tables, backend routes (Sakhi chat, nearby places, RevenueCat webhook).

**Done looks like**
- `lib/supabaseClient.ts` exports an initialised Supabase client using `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, with AsyncStorage session storage.
- `lib/auth.ts` exports `sendOtp`, `verifyOtp`, `signOut`, `getCurrentUser`, `onAuthStateChange`.
- `server/index.js` starts on `process.env.PORT`, binds `0.0.0.0`, loads secrets from env, and serves three routes: `POST /sakhi-chat`, `GET /nearby-places`, `POST /revenuecat-webhook`.
- `/sakhi-chat` verifies the Supabase Bearer token, checks the `profiles` row for `is_premium` + `sakhi_message_count`, enforces the 5-message free cap (returns 402 with `error:"limit_reached"` when exceeded), increments the count on success, and forwards the conversation to OpenAI `gpt-4o-mini` via a plain `fetch` call to `https://api.openai.com/v1/chat/completions` ‚Äî **do not add the `openai` npm package**, no SDK dependency needed.
- `/nearby-places` proxies Google Places Nearby Search (and `searchText` for shelter) using `GOOGLE_PLACES_API_KEY` with a strict field mask (`id,displayName,formattedAddress,location`); results cached 5 minutes in memory keyed by category + rounded lat/lng to **3 decimal places** (~111m precision ‚Äî 1 decimal place is too coarse at ~11km and would serve wrong-neighborhood results to nearby users).
- `/revenuecat-webhook` is guarded by `REVENUECAT_WEBHOOK_SECRET`; on event types `INITIAL_PURCHASE`, `RENEWAL`, `UNCANCELLATION`, `PRODUCT_CHANGE` sets `is_premium=true` and `premium_until` from `expiration_at_ms`; on `EXPIRATION` sets `is_premium=false`; returns 501 when the secret is not configured.
- The old `@workspace/api-client-react` import and `EXPO_PUBLIC_DOMAIN` / `setBaseUrl` wiring are removed from `app/_layout.tsx` and `app/(tabs)/sakhi.tsx`. `QueryClientProvider` and `@tanstack/react-query` are also removed if no other screen uses them.
- A `DATABASE_SETUP.sql` file is added at the repo root with the exact SQL the user should paste into the Supabase SQL Editor: `profiles` table (with auto-create trigger on `auth.users`), `live_sessions` table, and all RLS policies described in the spec.
- A README block lists the required Replit Secrets: `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `OPENAI_API_KEY`, `GOOGLE_PLACES_API_KEY`, `REVENUECAT_WEBHOOK_SECRET` (optional); and the required Expo public env vars: `EXPO_PUBLIC_SUPABASE_URL`, `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, `EXPO_PUBLIC_BACKEND_URL`, `EXPO_PUBLIC_LIVE_TRACKER_URL`.

**Out of scope**
Login UI screen (separate task). Map pins UI and live SOS UI (separate task). RevenueCat mobile SDK (separate task).

**Steps**

## 4 ‚Äî Real Map Pins + Live SOS Tracking

**What & Why**
Upgrade the map screen to show real nearby-place pins fetched from the backend (police stations, hospitals, pharmacies, shelters), and add live GPS tracking during SOS so trusted contacts receive a Supabase Realtime-powered tracking link instead of a static coordinate.

**Done looks like**
- `lib/nearbyPlaces.ts` exports `fetchNearbyPlaces(lat, lng, category)` ‚Äî fetches `${EXPO_PUBLIC_BACKEND_URL}/nearby-places`, returns a typed `NearbyPlace[]` (`id, name, address, lat, lng`).
- `app/(tabs)/map.tsx` (native only): after a category card is tapped, calls `fetchNearbyPlaces` and renders results as `<Marker>` pins on the existing `react-native-maps` `MapView`. **Each marker uses the same per-category color as its card icon** (police/hospital/pharmacy/shelter colors already defined in `CATEGORIES`) via `pinColor`. Tapping a marker opens a **`<Callout>`** (not a bottom sheet ‚Äî no new dependency needed) showing name, address, and a "Navigate" button that calls **`navigateTo(lat, lng, label)`** from `lib/native.ts` (the existing function that opens the device's maps app at exact coordinates ‚Äî distinct from the card-level `searchNearby`, which stays as the secondary category-search action on each card). Web platform (`MapPreview.web.tsx`) is unchanged.
- `lib/liveSession.ts` exports `startLiveSession()` (inserts a `live_sessions` row, returns `{ shareId, shareUrl }` where `shareUrl = ${EXPO_PUBLIC_LIVE_TRACKER_URL}/${shareId}`), `updateLiveSession(shareId, lat, lng, accuracy)`, and `endLiveSession(shareId)`. All calls are no-ops / silently skipped if no Supabase user is signed in.
- `context/SafetyContext.tsx` is updated: `triggerSOS()` calls `startLiveSession()` and begins `Location.watchPositionAsync` (high accuracy, ~10s interval, 10m distance filter), pushing updates via `updateLiveSession`. `cancelSOS()` (and provider unmount cleanup) stops the watcher and calls `endLiveSession`. `SosState` gains an optional `shareUrl` field populated once `startLiveSession` resolves.
- `components/SosModal.tsx` is updated: when `sos.shareUrl` is set, the SMS/WhatsApp message body uses it instead of the static Google Maps link. A small "Live tracking is on" badge with a green dot is shown in the location panel.
- `web/live.html` is a single self-contained HTML file (no build step): reads `shareId` from the URL path, connects to Supabase via the JS CDN client (Supabase URL and publishable key are hand-filled into the file before each deploy ‚Äî these are public-safe values protected by RLS, not secrets, but must be updated manually if keys ever rotate), loads the initial `live_sessions` row by `shareId`, subscribes to Realtime UPDATE events on that row, renders a moving marker on a Leaflet/OpenStreetMap map. Shows "Session ended" when `is_active` becomes false or `expires_at` is past. Works on any device without installing the app.

**Out of scope**
In-app real-time tracking of someone else's location (recipients use the web link). RevenueCat or Sakhi changes (separate tasks). Routing/turn-by-turn navigation (external maps app handles this).

Map screen ‚Äî fetch + pins ‚Äî Update `app/(tabs)/map.tsx`: on category tap, fetch nearby places; show a loading indicator on the card while fetching; render `<Marker>` pins colored per category, each with a `<Callout>` showing name, address, and a "Navigate" button wired to `navigateTo`.
