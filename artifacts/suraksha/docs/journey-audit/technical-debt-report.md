# 10. Remaining Technical Debt

## Carried forward / found during this audit

| ID | Debt | Impact | Why not fixed now |
|---|---|---|---|
| TD-1 | No server-side monitor for passed journey deadlines | The one residual reliability gap this pass cannot fully close from the mobile client alone (see Reliability Audit's "residual gap") — if the app is never reopened after the deadline passes, no auto-SOS can fire | Requires backend access this environment doesn't have. The `journeys` table now has the data a monitor would need (`started_at`, `duration_minutes`, `ended_at`) — this pass's repository work is the prerequisite, not the monitor itself. Recommended concretely: a scheduled job querying `journeys WHERE ended_at IS NULL AND started_at + (duration_minutes + grace_minutes) * interval '1 minute' < now()`, escalating the same way the client's auto-SOS does |
| TD-2 | A failed `journeyRepository.startJourney()` insert is never retried | The historical/monitoring record for that journey is permanently missing — not safety-critical (the local timer is unaffected), but it does mean a future backend monitor (TD-1) would have no row to alert on for that specific journey | Deliberately not built this pass — retry-with-offline-queue infrastructure exists for `sos_events` (safety-critical) but adding the same machinery for a non-safety-critical historical record is disproportionate until TD-1 (the thing that would actually depend on this data existing) is built |
| TD-3 | No `durationMinutes` range validation in `journeyRepository.startJourney()` | Not reachable through the current UI's fixed duration-preset picker; only relevant to a direct/malformed call | Low priority — RLS already prevents this from being anything other than a data-quality issue on the caller's own row, never a cross-user issue |
| TD-4 | No continuous route tracking / breadcrumb trail (`route_json` column exists in the schema, unused) | Journey is a time-based check-in, not a destination/route-based feature — this is a scope decision, not a defect, but it does mean several of this audit's requested review areas (geofencing, distance filter, speed/heading, adaptive GPS frequency) have no code to review | Building continuous route tracking is a substantial new feature (background GPS for the full journey duration, a destination picker, a geofence-based arrival detector) — explicitly out of scope per "do not redesign the architecture" / "do not build unrequested features." If this is genuinely wanted, it should follow the same "flag as a product decision, then implement deliberately" pattern used for biometric unlock and reauthentication in the auth-hardening passes, not be built silently as a side effect of a hardening audit |
| TD-5 | No `listForUser`/journey-history read path implemented | No UI currently needs one — `db.journeys.listForUser` exists at the DB-helper layer but has no domain-repository method or screen consuming it | Add a `JourneyRepository.listForUser()` method + screen only if/when a "past journeys" feature is actually requested — not built speculatively |
| TD-6 | Mock-location spoofing detection not implemented anywhere in the app (shared concern with SOS, not journey-specific) | A spoofed location could theoretically be embedded in a journey-start alert's location context, same as it already could in an SOS alert | Out of this audit's scope specifically — `core/permissions/location.ts`'s `getCurrentLocation()` is shared by both features; if this is pursued, it should be fixed once, centrally, not duplicated per-feature. Noted here for completeness since Section 3 of the brief asked about it directly |

## Debt paid down in this pass

1. **Zero repository-pattern wiring for journeys** — the `journeys` table existed with zero code ever calling it. Now has a full domain entity, repository interface, Supabase implementation, DTO/mapper, and DI registration.
2. **Silent, complete loss of the auto-SOS safety mechanism on any backgrounding** — the single most severe finding of this audit, fixed via wall-clock-derived status computation and crash/background recovery.
3. **No journey persistence across app kill/restart at all** — `journeyPersistence.ts` (new) closes this.
4. **Notification-rescheduling bug on resume** — the local "are you safe?" notification would have rescheduled a full-duration reminder measured from the resume moment (not the original deadline) once recovery was added; fixed by scheduling the *remaining* time from the wall-clock anchor instead.
5. **No journey-specific telemetry** — `core/analytics/journeyTelemetry.ts` (new) did not exist before this pass.
