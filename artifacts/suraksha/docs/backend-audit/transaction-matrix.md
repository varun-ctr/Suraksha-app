# 4. Transaction Matrix

Every single-table write in this schema (`INSERT ... RETURNING`, `UPDATE ... WHERE`, `UPSERT ... ON CONFLICT`) is already atomic — Postgres guarantees this per-statement with no explicit transaction needed. The matrix below is about **multi-statement operations**, where partial failure between statements can leave data in an inconsistent state.

| Operation | Statements involved | Currently atomic? | Risk if interrupted mid-way | Recommendation |
|---|---|---|---|---|
| SOS creation | 1: `INSERT sos_events` | ✅ Yes (single statement) | None | None needed |
| SOS alert dispatch | 1 backend call (`/sos/alert`, itself: idempotency-cache check → Twilio send → cache write, server-side) | ✅ Effectively — the idempotency cache is checked/written around the Twilio call, and a client retry with the same key returns the cached result rather than re-sending. Reviewed and confirmed solid in the SOS audit | Low — worst case is a cache-write failure after a successful Twilio send, which would allow one duplicate send on retry, not data loss | None needed — already the correct trade-off for an emergency alert (a possible duplicate SMS is far preferable to a possible missed one) |
| Journey creation | 1: `INSERT journeys` (client-generated UUID, retried with backoff — see journey audit v2) | ✅ Yes | None — retries are idempotent by exact id | None needed |
| Journey completion | 1: `UPDATE journeys SET ended_at = ...` | ✅ Yes | None | None needed |
| **Account deletion** | **6+ separate round trips**: delete contacts, resolve/delete `sos_events`, end `journeys`, end `live_sessions`, delete `notification_tokens`, delete/reset `subscriptions`, delete `profiles` row, plus the Firebase Auth user deletion itself and a backend call for anything server-owned | ❌ **No** — each is its own independent client-orchestrated request | **The one real gap in this matrix.** If the app is killed, loses network, or crashes partway through, some tables are cleaned up and others aren't — permanently orphaned rows with no database-level cascade to fall back on (see the ER Diagram's finding that all FKs to `auth.users` were dropped post-Firebase-migration) | **Backend recommendation** (not implementable from this environment): a single `SECURITY DEFINER` Postgres function, `public.delete_user_account(p_user_id TEXT)`, performing every table's delete inside one `BEGIN...COMMIT` block, callable via RPC from a verified-token backend route — so the whole account teardown either fully succeeds or fully rolls back, never partially applies. See the Backend Recommendations section of the Production Readiness Report for the full function sketch |
| Live session cleanup (zombie-session close on new SOS start) | 1: `UPDATE live_sessions SET is_active=false WHERE user_id=? AND is_active=true` (`endAllActiveForUser`) | ✅ Yes (single statement, closes all prior sessions for the user in one call) | None | None needed |
| Journey recovery (crash/background) | Local: read persisted state, compute status, `INSERT`/`UPDATE journeys`, trigger SOS if needed | ✅ Each individual write is atomic; the *sequence* isn't a single DB transaction, but each step is independently safe to retry/resume (idempotent by design — see the Journey audit's wall-clock recovery policy) | Low — designed from the ground up to tolerate interruption at any point | None needed — this is the correct pattern for a client-orchestrated recovery flow (a DB transaction can't span an app restart anyway) |

## Why account deletion is the one true P1/P2 transaction gap

Every other "multi-step" flow in this app is either (a) genuinely single-statement, or (b) explicitly designed to be interruption-tolerant via idempotent retries (SOS, journeys) — which is the *correct* answer for a mobile client that can be killed at any instant, not a workaround. Account deletion is different: it's a **one-time, user-initiated, high-consequence** operation where "partially succeeded" is a worse outcome than either fully succeeding or fully failing, and unlike SOS/journey creation, there's no safety reason to prefer partial completion over an atomic all-or-nothing operation. This is why it's flagged distinctly rather than graded on the same curve as the emergency-critical flows.
