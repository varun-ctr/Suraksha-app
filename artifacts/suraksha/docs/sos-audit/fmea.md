# 5. Failure Mode and Effects Analysis (FMEA)

Severity (S), Occurrence (O), Detection (D) scored 1 (best) – 5 (worst); RPN = S × O × D. Scope is the mobile-client SOS path only (backend `/sos/alert` already had its own idempotency/rate-limit hardening prior to this pass and is out of scope for re-scoring).

| Failure Mode | Effect | S | O | D | RPN | Current Mitigation | Residual Risk |
|---|---|---|---|---|---|---|---|
| Live tracking silently stops on backgrounding | Contacts see a frozen last-known location during an active emergency, believing it's current | 5 | 5 (was: every backgrounding event, pre-fix) | 4 (no signal it stopped) | **100 → 20** post-fix | Background location task (`expo-task-manager` + `startLocationUpdatesAsync`) keeps delivering after backgrounding; heartbeat (`expires_at`) makes a genuine stall detectable to viewers within 5 min instead of forever | Low — residual risk is OS-level background throttling under extreme low-power conditions, outside app control |
| `sos_events` write fails and is never retried | No durable record of the emergency; live tracking/alerts may still have fired, but the incident isn't in the user's history and any backend-side automation keyed off that table never triggers | 4 | 3 | 3 | 36 → **12** | 15s automatic retry while `phase === "active"`; offline-queue persists the pending write across a crash/kill so a relaunch can complete it | Low — remaining gap is the lack of a server-enforced idempotency constraint (see Technical Debt TD-1), so a race between a genuine second insert and a stale retry is theoretically possible in a narrow window |
| Duplicate `sos_events` row from a retried write | Two rows for one real emergency — cosmetic in the user's own history, but could double-count in any backend analytics/automation | 2 | 2 | 3 | 12 | `findRecentUnresolvedEvent` checks a 5-minute lookback window for an unresolved event before retrying an insert | Medium — best-effort only; no unique constraint exists at the DB layer to make this airtight (TD-1) |
| Zombie/duplicate live session from a crash mid-emergency | An old, no-longer-updating session stays `is_active = true` indefinitely, confusing anyone who still has the old share link | 3 | 3 (every unclean app exit during an active SOS, pre-fix) | 5 (nothing previously detected this) | 45 → **6** | `endAllActiveForUser` closes prior active sessions before a new one starts; `expires_at` heartbeat means even an undetected zombie naturally expires within 5 minutes via the `get_live_session` RPC's own expiry check | Low |
| SOS alert dispatched twice for the same trigger (crash recovery re-runs `dispatchAlerts`) | Contacts receive a duplicate SMS/call | 2 | 2 | 2 | 8 | `alertDispatchedRef` + persisted `alertsDispatched` flag in the offline queue gate re-dispatch; recovery reuses the same flag on resume | Low — a duplicate SMS is a minor nuisance, not a safety regression, and the directive explicitly prefers a possible duplicate over a missed alert |
| Empty contact list at trigger time | No one is alerted automatically | 5 | Depends entirely on onboarding completion, not a code defect | 1 (surfaced immediately, explicitly) | — | `SosBottomSheet` shows a dedicated "no contacts" banner + "Add Contacts" CTA; `callNumber("112")` remains available regardless | Low from a software standpoint — this is a data/onboarding risk, not a reliability defect; flagged in Risk Assessment |
| GPS/location permission denied or unavailable | Alert sent without location data | 3 | 2 | 1 (UI clearly shows "location unavailable") | 6 | Message builder omits the location line gracefully rather than sending a broken link; SOS still proceeds | Low |
| Backend and native SMS both fail (e.g. airplane mode) | No automatic contact delivery at all | 5 | 1 (requires simultaneous backend+cellular unavailability) | 3 | 15 | None beyond the manual "Call"/"SMS"/"WhatsApp" buttons already in `SosBottomSheet` as an always-available manual backup | Medium — this is a genuine device-connectivity limit rather than a software gap; see Risk Assessment for mitigation options (e.g. surfacing an explicit "alerts could not be sent — try calling" state, currently only implicit via per-contact `failed` badges) |
| Multiple/duplicate SOS triggers (double-tap, shake race) | Two overlapping SOS runs corrupt each other's state (GPS watch, live session, DB event) | 4 | 3 (easy to trigger accidentally) | 1 | 12 → **4** | `triggerSOS()` guard (`phase !== "idle"`) + `sosRunIdRef` staleness check on every async callback | Low |

## Highest-priority residual risk

**TD-1 (no server-enforced idempotency key on `sos_events`)** is the only item on this table whose current mitigation is explicitly best-effort rather than structurally guaranteed. It does not currently correspond to any known incident or reproducible failure — it's a defense-in-depth gap, not an active defect — and is tracked as a P1 in the Technical Debt Report with a concrete schema-migration fix recommended (a unique constraint on `(user_id, idempotency_key)`), which requires database migration access this environment does not have.
