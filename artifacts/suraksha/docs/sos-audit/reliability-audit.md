# 4. Reliability Audit — Failure Scenario Classification

Classification key:
- **Recoverable** — the system detects and resolves the failure itself, no data or delivery is lost, user may not even notice.
- **Retryable** — the system automatically retries and will eventually succeed once the underlying condition clears; a delay is possible but delivery is not lost.
- **Degraded** — the primary path fails but a fallback delivers a materially-weaker-but-real result (e.g. a manual SMS compose instead of automatic Twilio send).
- **Fatal** — no code path currently recovers from this; the emergency signal can be silently lost.
- **User-visible** — the UI reflects the failure state so the user isn't left assuming help is on the way when it isn't.

| # | Scenario | Classification | Where it's handled | Notes |
|---|---|---|---|---|
| 1 | No internet at trigger time | Retryable + User-visible | `insertSosEvent` fails → 15s retry loop (`SafetyContext.tsx`); `sosOfflineQueue` persists the pending activation so a later retry (or app relaunch once connectivity returns) still completes the write | The countdown/active UI never blocks on this — SOS still "activates" locally and the alert-dispatch/live-tracking paths proceed independently of whether the DB row exists yet |
| 2 | Poor/flaky internet (high latency, intermittent drops) | Retryable | `apiFetch` has a `timeoutMs` (10s for `/sos/alert`); `attemptBackendAlert` retries once after `BACKEND_RETRY_DELAY_MS` (2s) reusing the same idempotency key | A single retry is enough for a transient blip but not for a sustained multi-minute outage — see Technical Debt item TD-3 |
| 3 | GPS unavailable (no fix, indoors, cold start) | Degraded + User-visible | `getCurrentLocation()` in `core/permissions/location.ts` returns `null` on failure rather than throwing; `SosBottomSheet` shows "Getting your location…" / "Location unavailable" copy instead of blocking the SOS | The alert message (`buildEmergencyMessage`) gracefully omits the location line when `coords` is null rather than sending a broken link — contacts still get alerted, just without a location |
| 4 | Location permission denied | Degraded + User-visible | Same `null`-coords path as #3; the SOS flow still runs (countdown, DB event, contact alerts) without location data | The user should be prompted to re-enable location from Settings, but that redirect isn't currently wired — see Technical Debt |
| 5 | Backend `/sos/alert` timeout | Retryable + Degraded | One bounded retry (see #2); if both attempts fail, `sendSosAlerts` falls through to the native per-contact SMS compose + call-first-contact fallback unconditionally | Never fully fatal — the native fallback has no dependency on the backend being reachable at all |
| 6 | Firebase unavailable (auth token refresh fails) | Degraded | `canAttemptBackend()` checks `firebaseAuth.currentUser` before even trying the backend call, skipping straight to the native SMS/call fallback if there's no usable session | `insertSosEvent`/`startLiveSession` also require `firebaseAuth.currentUser` — if the user is fully signed out mid-emergency (should not normally happen, session persistence is encrypted-at-rest per the auth hardening pass) the DB write and live session are skipped, but native SMS/call still fire since they don't need Firebase |
| 7 | Supabase unavailable | Retryable | Covered by #1's retry loop for `sos_events`; `liveSessionRepository` calls are individually try/caught and logged, never thrown into the render path | Live tracking degrades to "no live link in the alert message" (falls back to a static Google Maps coordinate link) rather than blocking anything |
| 8 | Push/local notification delivery failure | Degraded | `scheduleLocalNotification`/`registerForPushNotifications` in `core/permissions/notifications.ts` swallow all failures and return a typed non-throwing result; the journey "are you safe?" check-in notification failing to schedule does not stop the overdue-timer logic that still runs the auto-SOS escalation purely on in-app timers | The overdue auto-trigger doesn't depend on the notification actually being delivered — it's a courtesy reminder, not the retry mechanism |
| 9 | App minimized (backgrounded, not killed) | Recoverable | Background location task (`core/permissions/backgroundLocation.ts`) keeps delivering location updates via the OS-level `Location.startLocationUpdatesAsync` task, independent of the JS/React lifecycle | This was the single biggest gap found in this audit before the fix — previously used `watchPositionAsync`, foreground-only, live tracking silently stopped within seconds of backgrounding |
| 10 | App killed outright (swiped away or crashed) mid-emergency | Recoverable + User-visible | Crash-recovery mount effect in `SafetyContext.tsx` reads `sosOfflineQueue.getPendingActivation()` on next launch and either resumes the active-SOS UI (if under 30 min old) or reconciles the DB record silently (if older) — see State Machine doc | Background location delivery itself can also relaunch the app in the background purely to deliver a location event, independent of the user reopening it, per iOS background-location semantics |
| 11 | Low battery / Low Power Mode | Degraded | `pausesUpdatesAutomatically: false` is explicitly set on the background location request so iOS Low Power Mode doesn't silently pause delivery the way the default would | iOS can still throttle background delivery frequency under extreme low-power conditions — this is an OS policy outside app control; the heartbeat/expiry mechanism (see #16) ensures a viewer at least learns tracking has gone stale rather than seeing a frozen pin with no indication anything's wrong |
| 12 | Airplane mode | Retryable (same as #1) | Identical code path to "no internet" — the DB write retry loop and offline queue don't distinguish the specific network condition, only "the request didn't succeed" | Native SMS/call fallback will also fail while airplane mode is on (no cellular radio); this is a genuine reachability limit, not a code gap — see Risk Assessment |
| 13 | Device reboot mid-emergency | Recoverable + User-visible | Same crash-recovery path as #10 — a reboot is indistinguishable from a kill from the app's perspective; `AsyncStorage`-persisted `PendingSosActivation` survives the reboot | The background location OS task itself does not survive a reboot until the app is relaunched at least once (standard OS behavior, not an app-level gap) |
| 14 | Memory pressure (OS kills the app to reclaim memory) | Recoverable | Identical to #10 — from the app's perspective this is just another process kill | |
| 15 | Location fetch timeout (GPS present but slow to acquire a fix) | Degraded + User-visible | `getCurrentLocation()` has its own internal timeout (returns `null` rather than hanging indefinitely); the countdown/active UI shows "Getting your location…" and proceeds to alert dispatch regardless of whether a fix ever arrives | The alert is never held hostage waiting for GPS — this was a deliberate existing design choice, confirmed correct against the "never sacrifice delivery for completeness" directive |
| 16 | Contacts list unavailable / empty | Fatal (by design, user-visible) | `sendSosAlerts` returns `[]` immediately if `contacts.length === 0`; `SosBottomSheet` shows a dedicated "no contacts" banner with a direct "Add Contacts" action | This is the one scenario genuinely un-recoverable in software — there is no one to alert. Correctly surfaced to the user rather than silently doing nothing; the native `callNumber("112")` action remains available regardless |
| 17 | Multiple rapid SOS taps / duplicate trigger requests (double-tap, shake + button, journey-overdue racing a manual tap) | Recoverable | `triggerSOS()` guards on `sos.phase !== "idle"` — every entry point (button, shake detector, journey-overdue auto-escalation) funnels through the same guarded function, so a second call while counting down or active is a no-op | `sosRunIdRef` additionally guards every async callback (location resolution, DB write, live-session creation) against acting on a stale, superseded run if a cancel-then-retrigger race does occur |

## Summary

Of the 17 scenarios: **6 recoverable, 5 retryable, 5 degraded-with-fallback, 1 fatal-by-design (empty contact list — correctly surfaced, not a code defect), 0 silently fatal.** No scenario in this table results in a real emergency being silently dropped with no user-visible indication and no automatic retry, which was not true before this pass (scenarios #9 and #10 were previously fatal-and-silent — see Technical Debt Report for the before/after).
