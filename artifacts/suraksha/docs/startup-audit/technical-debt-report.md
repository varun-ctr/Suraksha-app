# 7. Remaining Technical Debt

| ID | Debt | Impact | Why not fixed now |
|---|---|---|---|
| TD-1 | `expo-notifications` internally uses the same `requireNativeModule()`-throws-if-missing pattern as `expo-task-manager` did, across several of its own sub-modules (`NotificationScheduler`, `NotificationPresenterModule`, `BadgeModule`, etc.), and is imported unconditionally at `app/_layout.tsx`'s module scope | Structurally the same fragility class as the bug just fixed — theoretically, a runtime where one of these specific native sub-modules is missing would crash the bundle the same way. Has never manifested in practice: notifications are one of Expo's oldest, most universally-bundled modules (unlike TaskManager's background-execution surface, which Expo Go's own docs explicitly say is unsupported) | Notifications are a **required**, not optional, app capability — wrapping them in the same lazy/degrade-gracefully pattern used for TaskManager/Biometrics would mean the app silently has no push/local notifications in some runtime, which is a UX regression, not a safety improvement, for a capability that's supposed to always work. The correct mitigation already shipped: `installCrashBeforeRenderHandler()` means IF this ever throws during the startup window, it's now a `crash_before_render` telemetry event instead of a silent white screen — visible, not fixed. A full fix (capability-gating notifications) is out of scope for this pass and should only be pursued if telemetry ever actually shows this firing |
| TD-2 (fixed in this pass) | `SplashScreen.hideAsync()` was only ever called inside `Gate`, which never mounts when `!APP_CONFIG.ok` (the `ConfigErrorScreen` path) | The native splash screen could have remained visible on top of `ConfigErrorScreen` indefinitely on a misconfigured build | Fixed: `RootLayout` now has its own `useEffect` that calls `SplashScreen.hideAsync()` directly on the `!APP_CONFIG.ok` path, independent of whether `Gate` ever mounts |
| TD-3 | No real-device timing data exists yet for the Timing Report / Performance Score — both are estimates | Can't validate whether the 6-second auth timeout or 4-second font timeout are well-tuned for real-world conditions | Requires an actual device/EAS build to measure, which this sandboxed environment cannot produce. `core/analytics/startupTelemetry.ts` is now instrumented specifically so this becomes measurable in production without another audit pass |
| TD-4 | `core/permissions/notifications.ts`'s `enableNotificationHandler()` runs as a module-scope side effect (sets a JS-side notification handler at import time) | Very low risk (it's a synchronous JS callback registration, not a native module resolution), but it is technically a module-scope side effect, which Section 2 of this audit asked to eliminate entirely | Left as-is: this needs to run before any notification can arrive, and moving it into a component effect would risk a race where an early notification arrives before the handler is registered — the current approach is more correct for its actual purpose than the alternative, even though it's technically a "side effect at module scope." Documented here so it's a deliberate, reviewed exception, not an oversight |

## Debt paid down in this pass

1. The one confirmed, reproducing startup crash (expo-task-manager in Expo Go) — fixed via `core/capabilities/nativeCapabilities.ts`.
2. Biometrics' identical latent risk — fixed proactively before it was ever wired into a screen.
3. No structural safety net existed for a *future* mistake of the same shape — `installCrashBeforeRenderHandler()` now provides one.
4. All ad-hoc `[TEMP-DEBUG]` console logging — replaced with permanent, privacy-safe telemetry.
5. No visibility into real startup duration — `getElapsedSinceStart()` + the telemetry event set now provide it.
6. Splash screen could stay stuck over `ConfigErrorScreen` forever on a misconfigured build — fixed (TD-2).
